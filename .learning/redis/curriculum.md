# 커리큘럼: redis (COACH ONLY — 학습자 열람 금지)

## 관통 명제
Redis는 인메모리 자료구조 서버다 — 디스크 DB가 감당 못 하는 **고빈도 읽기의 오프로드(자원)** 와
**프로세스 밖 중앙 지점에서의 원자적 상태 갱신(정합성)** 을 판다. 그 대가로 시스템의 가용성 축이
하나 늘어난다(장애). → 사다리는 자원(S1, S3) · 정합성(S2) · 장애(S4)를 모두 포함 — §0 반려 규칙 통과.

## 도메인
이커머스 "요즘마켓" — 상품 카탈로그·리뷰·조회수·선착순 쿠폰·실시간 인기 랭킹. 전 스테이지 연속.

## 인프라 (고정, §7 — 2-1e 4축 도출)
- **인스턴스 수 축**: S2가 프로세스 간 조율(다중 인스턴스 oversell) → app 컨테이너 + nginx LB.
  단, compose `profiles: [cluster]`로 평시 비활성 — 평소 앱은 호스트 `./gradlew bootRun`(§7-1: 디버거/GC).
  S2부터 `docker compose --profile cluster up -d --build --scale app=3`, 테스트는 `BASE_URL=http://localhost:8088`.
- **관측 축**: S1·S3·S4의 멈춤/목격 조건이 부하 *중* 시계열(p99·커넥션 풀·에러율 추이) 요구
  → prometheus + grafana 포함, 스크레이프·대시보드 코치 완성 제공(끝). grafana http://localhost:3000.
- **자원 레버 축**: mysql `mem_limit: 512m / cpus: "1.0"`(§7-4, 온보딩부터 — 한도는 학습자에게 답 누설 ✗)
  + 데이터 볼륨(시드 스크립트 S1 진입 시 코치 제공) + vanilla 기본 한계(buffer pool 128MB·HikariCP 10).
- **장애 주입 축**: 장애원이 외부 API가 아니라 토픽 기술 자체 → 스텁 다운스트림 불요.
  주입은 `docker compose pause redis` / `unpause` CLI 한 줄 — compose 불변(§7-3).
- 토폴로지: mysql:8(캡) + redis:7(vanilla 1대, 답 설정 없음) + [cluster] app·nginx + prometheus + grafana. 고정.

## S0 (base) — 마켓 MVP
- 시나리오: 상품 등록/상세/가격수정 + 조회수 + 리뷰 요약. 대상 기술 없음, MySQL만.
- 인수 테스트: `app/src/test/java/harness/ProductApiSpec.java` (HTTP 행동 계약 — 구현 강제 없음)
- 심어둘 씨앗:
  ① 매 상세 조회 = product SELECT + 리뷰 집계(읽기 자원 씨앗 — 볼륨 커지면 폭발)
  ② 매 조회 view_count UPDATE = hot row 쓰기 경합(자원 씨앗)
  ③ "가격 수정 즉시 반영" 행동 계약 = 훗날 캐시 도입 시 stale/무효화 고통 예약
  ④ "조회수는 해당 조회 포함 누적" 계약 = 훗날 카운터 이동 시 계약 재협상 재료
- 완료: green (호스트 bootRun + compose의 mysql)

## S1 — 성장+라이브 트래픽에 상세 API가 죽는다  [타입: 자원]
- 도착 사건(유기적 성장+트래픽 사건): "오픈 4개월, 상품 30만·리뷰 300만 row. 다음 주 인플루언서
  라이브 커머스 공지 — 특정 상품에 트래픽 집중 예고." 아무 코드도 안 바꿨는데 아프기 시작.
- 1차 테스트(행동): 시드 투입 후 평시 부하(k6 저강도)에서 p99 베이스라인 기록 + ProductApiSpec 여전히 green.
- escalation(자원=부하 knob + 레버): k6 VU 단계 상승(hot key 편중 분포 — 상위 10개 상품에 80% 트래픽).
  레버 = ①mysql 캡(이미 배선) ②시드 볼륨(스크립트 코치 제공: 상품 30만·리뷰 300만·필요시 ×N)
  ③buffer pool 128MB 기본. 멈춤 조건: 부하 하 p99가 본인 베이스라인 ×5 이내 + 에러율 <1% + Spec green 유지.
- HIDDEN 솔루션: 읽기 캐싱(로컬 캐시든 redis든 — 증상 제거면 정답) + hot row UPDATE 오프로드(배치/지연 집계/
  외부 카운터 — 뭐든). 리뷰 요약 비정규화도 정답. **"redis" 단어 먼저 안 꺼냄.**
  가격 즉시 반영 계약이 캐시와 긴장 → TTL/무효화를 스스로 설계하게 됨. viewCount 엄밀 계약이 부담이면
  학습자가 제기할 때 PM 자격으로 계약 완화 티켓 발행 가능("조회수는 1분 내 반영이면 됨") — 테스트 수정은 코치가.
- 그릴링: 왜 빨라졌나(무엇이 사라졌나) / 이 캐시는 어떤 조건에서 틀린 값을 보여주나 / 뭘 버렸나(신선도).
- 졸업 인식: 멈춤 조건 충족 + stale 허용 범위를 말로 설명 가능.
- 분기 메모: 로컬(인메모리) 캐시로 풀면 → S2 도착 사건(3대 증설)에서 "인스턴스마다 가격 다름" stale이
  쿠폰과 함께 터지도록 배치. redis 직행도 정당 — 그 경우 S2는 쿠폰 정합성만.

## S2 — 선착순 쿠폰 oversell + 3대 증설  [타입: 정합성]
- 도착 사건(토폴로지 변화 + 새 기능 티켓): "라이브 트래픽 겪고 가용성 때문에 앱 3대 증설 결정(cluster
  profile 가동). 동시에 기획 티켓: '선착순 100명 20% 쿠폰'." 티켓은 행동만: 100명 한정, 초과 발급 절대 금지,
  발급 여부 유저가 확인 가능.
- 1차 테스트(행동): CouponGrantSpec — 재고 100 + **동시 200 요청**(결정적, 부하테스트 ✗) → 발급 성공 정확히 100 ·
  총발급 ≤ 100 불변식 · 실패 요청은 명시적 거절. 단일 인스턴스에서 naive check-then-act가 oversell로 즉사.
- escalation(정합성=결정적 동시성 + --scale):
  ① 학습자 fix 후 `--scale app=3` + nginx 경유 재실행 → synchronized/로컬 락이면 재살해(§7-3 예시 그대로).
  ② DB 락(FOR UPDATE 등)으로 green이면 정당 — 이어서 [자원 전환] 사건: "쿠폰 오픈 순간 상세 조회 부하 동시"
     (k6 혼합 시나리오) → HikariCP 기본 10 고갈, 쿠폰 때문에 쇼핑몰 전체 p99 폭발("쿠폰 열었더니 몰이 죽어요").
  ③ S1을 로컬 캐시로 푼 학습자면 여기서 가격 stale CS 사건 동시 투입("인스턴스마다 가격이 달라요").
- HIDDEN 솔루션: 중앙 원자 연산(INCR/DECR/SETNX/Lua) 또는 직렬화 큐 — 불변식+SLO 증상 제거면 뭐든 정답.
- 그릴링: 3대에서 DB 락은 왜 되고 synchronized는 왜 안 되나 / 지금 그 중앙 지점이 죽으면?(S4 예약, 답 안 줌)
  / 뭘 버렸나(강한 일관성 vs 처리량, 발급 기록의 진실은 어디).
- 졸업 인식: 3대 + 혼합 부하에서 불변식 green + 상세 p99 유지.

## S3 — "지금 뜨는 상품 TOP10" 실시간 랭킹  [타입: 자원]
- 도착 사건(새 기능 티켓): "메인 개편 — '지금 뜨는 상품 TOP10'(최근 1시간 조회 기준, 준실시간) 노출.
  기획: '어제 순위 말고 지금요.'"
- 1차 테스트(행동): RankingSpec — 조회 이벤트 발생시킨 뒤 TOP10 순서가 최근 1시간 조회량 순 · 메인 API 응답.
  naive(이벤트 로그 테이블 + 시간윈도우 GROUP BY)로 소량에선 green.
- escalation(자원=부하 knob + 볼륨 레버): 조회 이벤트 초당 수백(k6) + 메인 페이지 rps 동시 → 로그 적재와
  집계 쿼리 경합, 랭킹 API p99 폭발 + 상세까지 전이. 레버 = 이벤트 볼륨(시드 ×N) + mysql 캡.
  멈춤: 부하 하 랭킹 p99 상대 목표 + 랭킹 반영 지연 ≤ 60s(행동 계약).
- HIDDEN 솔루션: 인메모리 순위 자료구조(sorted set) / 주기 스냅샷 / 근사 집계 — 증상 제거면 뭐든.
- 그릴링: 정확성 vs 실시간성 뭘 버렸나 / 1시간 윈도우를 어떻게 흘려보내나 / 재시작하면 랭킹은?
- 졸업 인식: 멈춤 조건 충족 + 윈도우/휘발성 트레이드오프 언어화.

## S4 — 새벽 redis 다운 15분, 전면 장애  [타입: 장애]
- 도착 사건(외부 변화/사고): 이 시점 학습자 시스템은 중앙 인메모리 저장소에 강결합돼 있다.
  "새벽 2시 그 프로세스가 15분 죽었다(코치가 `docker compose pause redis`). 전면 500. 복구 후에도
  5분간 콜드 캐시로 DB 폭격 → 2차 사망. 아침 회의: '캐시 좋자고 넣은 게 왜 SPOF가 됐죠?'"
- 1차 테스트(행동): DegradationSpec — redis pause 상태에서 ①상세 조회는 응답(열화 허용, 에러율 SLO 내)
  ②쿠폰 발급은 **오발급 0 유지**(불변식 — 명시적 실패는 허용) ③랭킹은 명시된 fallback 행동.
- escalation(장애=주입 knob): pause 시간 연장 → unpause 직후 k6 부하(콜드 캐시 stampede 목격, grafana로
  DB 커넥션·p99 추이 관찰) → 반복 pause/unpause(플래핑).
- HIDDEN 솔루션: 타임아웃/fallback/서킷브레이커/stampede 완화(TTL jitter·단일 재적재·워밍) — 증상 제거면 뭐든.
- 그릴링: 캐시는 이제 시스템에서 뭐가 됐나(가속기→가용성 축) / 쿠폰 카운터의 진실은 다운 중 어디에 있나 /
  뭘 버렸나(신선도 vs 생존).
- 졸업 인식: 주입+플래핑 하에서 SLO·불변식 green.

## 졸업 신호
S4 통과 + synthesis.md에 S1~S4 결정 로그(문제→시도→깨짐→선택→트레이드오프) 누적.

## 코치 리마인더
- 시드 스크립트·k6 스크립트는 해당 스테이지 진입 시 완성 제공(측정 도구 = 낚싯대, §4).
- 어떤 스테이지든 학습자 fix가 로컬에서 안 깨지면 "불필요 졸업" 인정 + progress에 기록 + 남은 escalation 타입 전환 재설계(§4).
- assert는 유저/도메인 결과만. 내부 지표(히트율·쿼리 수)는 진단 출력으로만.
